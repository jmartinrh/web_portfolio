Pacing & Optimization

Context

D&D Zombie Dragon Adventure was the studio's most ambitious UEFN project. The goal was to deliver a roguelike with 50 non-linear, interconnected rooms across 5 distinct biomes with backtracking, and open sightlines. It was 2x larger than anything we had built before. As the main level designer on the project I was responsible for designing and implementing all 50 rooms.

Problem

From my previous work with UEFN islands, it became clear that the planned map layout and asset density would not fit within UEFN's memory constraints. Using data layers was out of the question, as these are replicated in multiplayer games. The non-linear map meant players could take different routes and a complex system of conditions would be needed to load and unload portions of the map correctly. This had a high risk of creating soft-locks or a lot of edge cases to consider.

Goal

Design a level architecture and optimization pipeline that would deliver the 50-room scope within memory budget, without compromising the open, non-linear feel that we wanted to achieve with this game.

Solution

UEFN has a fairly easy to use World Partition tool, and this was going to be the main avenue of optimization I was going to employ. From previous optimization challenges the studio had faced, I knew the best course of action would be to divide the map into different sections and have the player only load one section at a time. Moving one player from one section to the next would require a teleport and probably a short loading screen.

The game was already divided into 5 biomes, so this was our starting point. With some experimentation and support from Tech Art, we figured sections of 10 to 12 rooms would support the asset density we were aiming for. Each section would have connections to others where players would be teleported.

For the setup to work optimally, the sections would have to be in their own World Partition cell with all adjacent cells completely empty of content. This way, even if the player reached the edge of a section/cell, they would not load any additional assets. Only the first section of the game, which was done very early, couldn't be spaced out optimally.

Here's how the map ended up looking from above with each area in its own cell.

One of the original ideas for the game was to create moments where players could see the places they would eventually reach, making the whole map feel truly connected. However, long sightlines and loading faraway assets would create its own set of performance problems. For this reason, I had to redesign a lot of the original level layouts. I had to rework the 2D layouts another designer had delivered and redo interroom connections to hide asset loading wherever I could.

Teleporting in Fortnite is near instantaneous no matter the platform you are playing on. This is great in current gen consoles as assets don't take long to load. But on older consoles there can be a lot of uncomfortable pop-in as the world loads around the player piece-by-piece. Thus, the game needed a way to hide asset loading.

We already had a simple teleport loading screen module implemented, but since there is no way to know if a player is using a low or high end platform, loading screen times would be the same across all players. This created a lot of unnecessary friction in current gen devices, so we needed to come up with something else to help hide loading.

In early prototypes I had presented a few navigational challenges (parkour, ziplines, grindrails, skydives) as ways to add variety in player movement from one room to the next. Taking a cue from what big single player games often do to obscure their loading screens, I proposed we used these navigation challenges to create some of the transitions between biomes. These transitions, then, would serve three purposes: break up pacing, communicate a transition, and hide asset loading.

To further hide level-loading, I made the assets around the players' teleport endpoints to be always loaded. It was key not to overdo this, but I made sure that the immediate surrounding after the player was teleported had basic structural assets (ground, walls, big cover pieces). This way I avoided them getting disoriented in the empty void of still-loading assets with invisible collisions.

Result

The level design architecture held — players responded well to the layouts with no meaningful navigation complaints. In the end, delivering 51 rooms, 6 rest zones, 3 boss arenas, and 1 hub was only possible because game launch was delayed two months. The game still had performance issues at launch but most were rooted in logic density, something level design alone wouldn't solve. These were solved post launch.
